# 验证与发布中文契约

## 一、硬失败

以下任一情况必须中止：NPK/IMG 不可读、内部路径重复或意外、版本不支持或路由错误、非 LINK 帧不可解码、LINK 无效、元数据越界变化、格式/载荷混合、DDS/BC 非法、意外空帧/黑帧、未声明 BGRA 差异、缺少哈希或报告。

## 二、独立检查

至少使用一个不共享生成器解析器的路径。最低检查 NPK magic、entry count、头部哈希、路径解密与唯一性、条目边界、IMG magic 和 version；结构变化越深，独立路径也必须越深。

## 三、视觉检查

- 解码所有非 LINK 帧。
- 生成覆盖全部帧的黑、白、棋盘联系表。
- 对未修改 BGRA 做逐字节基线比较。
- 透明和黑色状态按源语义解释，不使用通用阈值武断删除。

## 四、发布报告固定字段

1. 时间与模式。
2. 源包和基线哈希。
3. 生成器/验证器版本与哈希。
4. 输出路径、大小和 SHA-256。
5. IMG/帧/版本/格式数量。
6. 允许、实际、排除、缺失和意外变化。
7. 验证产物与异常。
8. 部署目标与回滚路径。
9. 待目标客户端验证项。

逐帧重绘任务还必须记录 source inventory、redraw run plan、逐帧 attempt、模型/适配器/包装器 provenance、generated/edited/runtime 哈希清单、Photoshop 适配说明、配置漂移计数和时序验收结果。

## 五、两阶段发布闭环

1. 预发布资源计划保持 `fullSkillCoverageProven=false`。
2. 使用新的版本化产物和新的空验证目录运行最终门禁；final summary 生成后保持不可变。
3. final summary 只声明是否允许生成发布元数据，不自行改写 manifest 或 `release.json`。
4. 从通过的 summary 更新两份元数据后，运行 `tools/Test-DnfReleaseClosure.ps1` 校验三方路径、长度、哈希、工具 provenance、实时独立索引和未部署状态。
5. 更新 README 后运行 `tools/Test-DnfProjectGate.ps1`。工具或来源哈希在 summary 之后变化时必须新建验证目录重跑。

## 六、部署

当前请求必须明确授权部署。先确认目标与冲突，再备份、使用非 NPK 临时名暂存、校验哈希、原子替换、独立复核并清理本次临时文件。

不得从文件名前缀推断加载优先级，也不得承诺检测规避或账号安全。

不得随机化、伪造或回填时间戳、签名或来源；压缩、加密、封装参数和工具 round-trip 均不能作为检测规避或客户端兼容证明。
